Assignment 1 (Autumn 2022)

Please read the whole page word by word from top to bottom at least once. Do not just skim through. Before you ask a question please check again whether the answer can already be found here and give yourself some time to think.

Context

We consider a game that works as follows:

  • At each round the player is given some digits.
  • The player submits a sequence of digits using some or all of the given digits for the round.
  • The submitted sequence must be one of the allowed ones as specified in a given list. The list may change from game instance to game instance but stays the same across different rounds of the same game instance.
  • The sequence submitted by the player at each round gets a score that is computed as follows:
    • For each digit, the corresponding value is added (however values may be positive or negative). A list is given with the value for each digit. This list may change from game instance to game instance but stays the same across different rounds of the same game instance.
    • The value of (sub)sequences of digits in consecutive ascending order are multiplied times the length of the sequence.
  • The player clearly wants to submit the sequence with the highest possible score for that round.

In this example the player is given the following digits during a round: 

01700

Let's see what can be played and what's the associated score:

  • 476234 can't be played because, among other things, the player doesn't have a 4.
  • 017 can be played. The value for 0 is 2, for 1 it's 4, for 7 it's -10. Moreover, 0 and 1 are a (sub)sequence of digits in consecutive ascending order of length 2. Therefore the score is: ((2 + 4) * 2) - 10 = 2.
  • 70 can be played with score -10 + 2 = -8.
  • 100 can be played with score 4 + 2 + 2 = 8.
  • 770 can't be played because the player has only one 7.
  • 00 can be played with score 2 + 2 = 4.
  • 3443 can't be played because, among other things, the player doesn't have a 3.
  • 01 can be played and the score is (2 + 4) * 2 = 12.
  • 1268 can't be played because, among other things, the player doesn't have a 2.
  • 7001 can be played and the score is -10 + 2 + (2 + 4) * 2 = 4.
  • 111111 can't be played because the player doesn't have these many 1.

Therefore for this round the player should submit the sequence 01.

The program

Write a program which:

  • Reads from the user:
    • The name of a text file containing the allowed sequences.
    • The name of a text file containing the values associated to each digit.
    • The name of a text file containing the digits given for each round.
    • The name of a text file for output.
  • Reads and suitably stores the content of the input text files.
  • Computes the best sequence for each round and the associated score.
  • Prints the result on a text file (see example below).

The program itself mustn't print anything on the terminal and just reads the name of the text files in the specified order (this is important, the testing won't work otherwise). Therefore the interaction between the user and the program would look like this (these are written by the user, not printed by the program):

Once again it's important that the format is exactly like this, on each line: the given digits for that round, the sequence played, the score. Otherwise the testing will not work.

As you can see in this example the “best sequence” for a round may be one with a negative score.

Assume that the overall input is such that at least one of the allowed sequences can always be submitted with the digits given at a certain round.

Functions

The following functions need to be included and meaningfully used in the program. You are given the implementation for the first one, the other ones need to be implemented.

Do not alter the function interface (i.e. the first line provided) of the functions in any way as the testing will not work otherwise.

The functions also need to work exactly as described as they will be tested one by one too.

You can define and use also other functions.

Conversion

Use the following function to convert from std::string to an std::vector of int: 

std::vector<int> string_to_digits(std::string s){
 
    std::vector<int> out; 
 
    for(int i = 0; i < s.size(); i++){
 
       std::string digit_string;
       digit_string.push_back(s[i]);
 
       int digit_int = std::stoi(digit_string);
 
       out.push_back(digit_int);
 
    }
 
    return out;
 
}

Possible sequence from digits

Implement and (meaningfully) use this function to determine whether a certain sequence can be generated using the given digits. 

bool possible_sequence(std::string digits, std::string sequence)

For example:

  • If digits is 01700 and sequence is 770 the function must return false.
  • If digits is 01700 and sequence is 7001 the function must return true.

Score

Implement and (meaningfully) use this function to determine the score associated to a sequence. 

int score(std::string s, const std::vector<int>& values)

For example:

  • If s is 7001 and values is 2, 4, -1, 5, -6, 1, 0, -10, 100, -3 the function must return 4.
  • If s is 14568010 and values is 2, 4, -1, 5, -6, 1, 0, -10, 100, -3 the function must return 103.
  • If s is 035 and values is 2, 4, -1, 5, -6, 1, 0, -10, 100, -3 the function must return 8.
  • If s is 04 and values is 2, 4, -1, 5, -6, 1, 0, -10, 100, -3 the function must return -4.
  • If s is 45601 and values is 2, 4, -1, 5, -6, 1, 0, -10, 100, -3 the function must return -3.

Best sequence

Implement and (meaningfully) use this function to determine the best sequence that can be submitted on the basis of the given digits for a certain round and of the information about allowed sequences and digit values. 

int get_best_sequence(std::string digits, const std::vector<std::string>& allowed, const std::vector<int>& values, std::string& best_sequence)

For example if we consider that digits is 01700, that allowed has the content of allowed.txt (see relevant example above) and that values has the content of values.txt (see relevant example above), then the function must provide in output 01 in best_sequence (output parameter passed by reference) and it must return 12.

Guidelines

Do not add any user or file input or output other than what has been prescribed, otherwise this will interfere with the testing. Remember to remove for example any prints used for debugging before the submission.

All the variables should be declared in the scope of a function (either the main or some other one). In other words, global variables are not allowed.

All the loops should be controlled/terminated either by the loop condition or by return. Statements such as break, continue, goto are not allowed anywhere in the program. Do not use the switch statement.

Do not use (functions from) the algorithm header.

You can only use headers from the C++ standard library.

The code must be such that it successfully compiles (using the default configuration) using as compiler gcc/g++ (9.4 or 10.3) or clang (12.0.1). If you can't test with these compilers and you want to be sure that there are no issues, I would advise you to only use features that have been covered during the module.

Assessment criteria

In order to be awarded a mark in the the 70%-100% range, a submission must fulfill all of these conditions:

  • All the requirements and guidelines must be respected.
  • The code must be entirely correct. For example (this is not an exhaustive list):
    • If one or more of the automated tests are not passed, the code is not correct.
    • If there are occurrences of crashes, infinite loops, or infinite recursion, the code is not correct.
    • If there are undefined behaviour instructions in the code, then it is not correct. This is regardless of whether or not it is noticeable when the program is run and regardless of whether it seems to work on your computer.
  • The code must be generally well indented and formatted and readable, with reasonably meaningful names for the variables.

Within the 70%-100% range the criteria are:

  • The more readable the code, the better.
  • Good functional decomposition should be applied, also in order to avoid code duplication.
  • The code doesn't need to be optimised for efficiency, but wastefulness should be avoided, as in the examples in the notes. Think of the prime number example, in which the loop terminates as soon as the first factor is found (instead of going on to try other potential factors) or as soon as the square root of the number is reached (instead of trying all the numbers up to the input).

You can assume that the functions and the program will always be tested with meaningful and consistent input.

We are not really going to consider comments in this submission. Feel free to add some if you think they would be useful for yourself to understand your own code or to explain some aspects of your program. But it is more important that your code is clear and readable (minimising the need for comments). Do not add too many comments as that can actually make the code less readable.

Plagiarism and collusion

I remind you that this is an individual assignment and you are expected to submit only your own work, as during an exam. Thus, you are expected not to submit work based on someone else's, not to show your work to others and not to work in collaboration with others.

The submissions will be checked for plagiarism and collusion using a detection tool. Please do not include any personal or identifiable information in the content of your source file (such as writing your name or username in comments).

It's important that your code for assessed assignments is never on any public repositories. This implies that you can't upload it on, for example, a public github repository (not even when you have completed first year) and it can't be on a public replit account (all free replit accounts are public).
